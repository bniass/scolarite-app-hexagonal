3. Flux Outbox Pattern (événement métier → Kafka)

Garantit la cohérence transactionnelle entre l'écriture en base et la publication des événements.

    participant K as Kafka

    rect rgba(37, 99, 235, 0.08)

    U-->>W: AnneeAcademique
    W-->>C: 201 Created

    loop polling
      P->>O: findPending()
      P->>K: send(Avro(CreateAnneeAcademiqueAvroModel))
      K-->>P: ack
      P->>O: markAsSent()
    end

4. Machine à états de AnneeAcademique

Implémentation du State Pattern dans domain/state/ : EtatAnnee (interface), AbstractEtatAnnee (refuse tout par défaut), et 5 états concrets. Toute transition non autorisée lève AnneeAcademiqueException.

stateDiagram-v2
    direction LR
    [*] --> AnneeBrouillon : creer(annee, datesAnnee)
    AnneeBrouillon --> AnneeBrouillon : modifier(dates)
    AnneeBrouillon --> AnneePubliee : publier()
    AnneePubliee --> InscriptionsOuvertes : ouvrirInscription()
    InscriptionsOuvertes --> InscriptionsFermees : cloturerInscription()
    InscriptionsFermees --> AnneeCloturee : cloturer()
    AnneeCloturee --> [*]

    note right of AnneeBrouillon
      modifier() ré-initialise
      dates + mois académiques
    end note

    note right of AnneePubliee
      datePublication = now()
    end note

    note right of AnneeCloturee
      État terminal
    end note

Matrice des transitions

État \ Action modifier publier ouvrirInscriptions fermerInscriptions cloturer
AnneeBrouillonreste Brouillon→ Publieeinterditinterditinterdit
AnneePublieeinterditinterdit→ InscriptionsOuvertesinterditinterdit
InscriptionsOuvertesinterditinterditinterdit→ InscriptionsFermeesinterdit
InscriptionsFermeesinterditinterditinterditinterdit→ AnneeCloturee
AnneeClotureeétat terminal — toutes les actions interdites

Observations

  • Dans AnneeAcademique.modifier(), setData() force l'état à new AnneeBrouillon() avant d'appeler etatAnnee.modifier(...). La garde du State Pattern est donc court-circuitée : un modifier() depuis n'importe quel état réinitialise à Brouillon au lieu de lever une exception.
  • Toutes les méthodes publiques émettent un AnneeAcademiqueCreeeEvent (avec un statut différent), alors que des classes d'événement dédiées existent (AnneeAcademiquePublieeEvent, InscriptionsOuvertesEvent, etc.). La typologie gagnerait à être homogénéisée.

5. Diagramme de séquence — Création d'une année académique

Flux complet du POST /api/academic-years : phase transactionnelle synchrone (Controller → UseCase → Domain → Repository MySQL → Outbox), puis publication Kafka asynchrone via OutboxPublisher @Scheduled(5s).

sequenceDiagram
    autonumber
    actor Client
    participant Ctrl as AnneeAcademiqueController
    participant Mapper as AnneeAcademiqueMapper
    participant UC as CreerAnneeAcademiqueService
@Transactional
    participant Repo as Repository
(MySqlAdapter)
    participant Domain as AnneeAcademique
    participant Outbox as OutboxPort
    participant DB as MySQL
    participant Pub as OutboxPublisher
@Scheduled(5s)
    participant K as Kafka + Schema Registry

    Client->>Ctrl: POST /api/academic-years
CreerAnneeAcademiqueRequest
    Note over Ctrl: @Pattern("^[0-9]{4}$")
    Ctrl->>Mapper: toCommand(request)
    Mapper-->>Ctrl: CreerAnneeAcademiqueCommand
    Ctrl->>UC: executer(cmd)

    rect rgba(37, 99, 235, 0.08)
      Note over UC,DB: Transaction unique

      UC->>Repo: findByCode(cmd.codeAnnee)
      Repo->>DB: SELECT par code
      DB-->>Repo: Optional
      Repo-->>UC: Optional

      alt déjà présente
        UC-->>Ctrl: AnneeAcademiqueException
        Ctrl-->>Client: 4xx (GlobalControllerAdvice)
      else nouvelle
        UC->>UC: new DatesAnnee(...)
        UC->>Domain: AnneeAcademique.creer(code, dates)
        activate Domain
        Domain->>Domain: checkDates()
        Domain->>Domain: checkIfDureAnnneEstValide(9 mois)
        Domain->>Domain: genererMoisAcademiques()
        Domain->>Domain: changerEtat(AnneeBrouillon)
        Domain->>Domain: addEvent(AnneeAcademiqueCreeeEvent)
        Domain-->>UC: AnneeAcademique
        deactivate Domain

        UC->>Repo: save(annee)
        Repo->>DB: INSERT annee_academique + mois_academique
        DB-->>Repo: ok

        UC->>Domain: pullDomainEvents()
        Domain-->>UC: List

        loop pour chaque event
          UC->>Outbox: save(event)
          Outbox->>DB: INSERT outbox_event
(payload Avro, status=PENDING)
        end
      end
    end

    UC-->>Ctrl: void
    Ctrl-->>Client: 201 Created + Location

    Note over Pub,K: --- Publication asynchrone (toutes les 5s) ---

    loop @Scheduled(fixedDelay = 5000)
      Pub->>DB: findTop50(PENDING, FAILED)
      DB-->>Pub: events
      loop pour chaque event
        Pub->>Pub: AvroSerializerUtil.fromBytes(payload)
        Pub->>K: send(topic, aggregateId, AvroModel)
        alt succès
          K-->>Pub: ack
          Pub->>DB: status = PUBLISHED
        else échec
          Pub->>DB: status = FAILED
        end
      end
    end

Garanties

  • Atomicité persistance + outbox — la même transaction couvre save(annee) et outboxPort.save(event).
  • Au moins une livraison Kafka — polling périodique + retry des FAILED.
  • Ordre par agrégat — ORDER BY occurredAt ASC et clé Kafka = aggregateId (même partition).
  • Idempotence côté consommateur — à assurer en aval.

Observation

Le bean DomainEventPublisher est injecté dans CreerAnneeAcademiqueService mais jamais utilisé — c'est outboxPort::save qui est appelé directement. Il y a deux chemins concurrents vers la table outbox, l'un est mort. À nettoyer.
